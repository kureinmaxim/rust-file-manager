# GUIDE: как устроен xtask (разбор для начинающего)

Этот документ объясняет, **как работает** инструмент `cargo xtask` в этом проекте, **почему он написан именно так** и какие конструкции Rust в нём используются. Читать вместе с кодом: [`xtask/src/main.rs`](xtask/src/main.rs).

---

## 1. Что такое xtask-паттерн и зачем он нужен

Каждому проекту нужны вспомогательные скрипты: поднять версию, что-то сгенерировать, проверить синхронизацию файлов. Классические варианты — bash или Python — имеют недостатки:

- **bash** не переносим (на Windows не работает без WSL) и легко ломается на краевых случаях;
- **Python** требует установленного интерпретатора и виртуального окружения — лишняя зависимость для Rust-проекта;
- оба не проверяются компилятором: опечатка обнаружится только при запуске.

**xtask** — это соглашение в Rust-сообществе: вспомогательные задачи пишутся **на самом Rust** как обычная программа внутри проекта. Получаем:

- один язык во всём проекте — изучая инструмент, изучаешь Rust;
- компилятор ловит ошибки до запуска;
- работает везде, где работает `cargo` — ничего дополнительно ставить не надо.

## 2. Из чего состоит механизм (3 детали)

### Деталь 1: workspace в корневом `Cargo.toml`

```toml
[workspace]
members = ["xtask"]
```

**Workspace** — это «зонтик» над несколькими пакетами (crate): они делят общий `Cargo.lock` и общую папку `target/`. У нас два пакета: основное приложение (корень) и `xtask/`. Благодаря этому `cargo` умеет собирать и запускать `xtask` прямо из проекта.

### Деталь 2: сам пакет `xtask/`

```toml
[package]
name = "xtask"
publish = false      # никогда не публиковать на crates.io — это внутренний инструмент

[dependencies]
                     # пусто! только стандартная библиотека
```

`publish = false` — страховка: даже случайно этот пакет не уедет в публичный реестр. Пустые зависимости — осознанное решение: для разбора пары строк TOML не нужна целая библиотека, а каждая зависимость — это время компиляции и поверхность для проблем.

### Деталь 3: алиас в `.cargo/config.toml`

```toml
[alias]
xtask = "run -q -p xtask --"
```

Когда вы печатаете `cargo xtask status`, cargo разворачивает это в `cargo run -q -p xtask -- status`:
- `run` — собрать и запустить;
- `-q` (quiet) — не печатать сообщения о компиляции;
- `-p xtask` — запустить именно пакет `xtask`, а не основное приложение;
- `--` — всё, что после, передать программе как её аргументы.

То есть «магии» нет: это обычная Rust-программа, просто удобно вызываемая.

## 3. Разбор кода `xtask/src/main.rs`

### 3.1. Точка входа: `main` и разбор аргументов

```rust
fn main() {
    let args: Vec<String> = std::env::args().skip(1).collect();
    let args: Vec<&str> = args.iter().map(String::as_str).collect();
    match args.as_slice() {
        ["status"] => status(),
        ["set", version] => { sync(Some(version)); }
        ["bump", level] => bump(level),
        _ => print_help(),
    }
}
```

Что здесь происходит, строка за строкой:

- `std::env::args()` возвращает **итератор** по аргументам командной строки. Нулевой аргумент — путь к самой программе, он нам не нужен, поэтому `.skip(1)`.
- `.collect()` собирает итератор в вектор `Vec<String>`.
- Вторая строка превращает `Vec<String>` в `Vec<&str>` (ссылки на строки). Зачем? Чтобы в `match` можно было писать литералы `"status"` — сравнивать `&str` с `&str` проще, чем со `String`.
- `match args.as_slice()` — самое интересное. Это **сопоставление с образцом** (pattern matching) по срезу: образец `["set", version]` совпадает только если аргументов ровно два и первый — `"set"`; при этом второй аргумент **автоматически попадает в переменную** `version`. Никаких `if args.len() == 2 && args[0] == "set"` — компилятор сам проверит все случаи.
- `_` — образец «всё остальное»: любые нераспознанные аргументы ведут к справке. Это правильное поведение CLI: не падать, а подсказать.

> **Почему так правильно:** match по срезу — идиоматичный Rust-способ разбора простых CLI без библиотек. Для больших CLI берут библиотеку `clap`, но для пяти команд это стрельба из пушки по воробьям.

### 3.2. Где корень проекта: `workspace_root`

```rust
fn workspace_root() -> PathBuf {
    Path::new(env!("CARGO_MANIFEST_DIR"))
        .parent()
        .expect("xtask must live inside the workspace")
        .to_path_buf()
}
```

- `env!("CARGO_MANIFEST_DIR")` — это **макрос времени компиляции** (восклицательный знак — признак макроса). Cargo при сборке подставляет сюда путь к папке `xtask/` прямо в текст программы. Поэтому инструмент работает из любого подкаталога — он не зависит от того, откуда его запустили.
- `.parent()` поднимается на уровень выше (`xtask/` → корень проекта) и возвращает `Option<&Path>` — «может быть путь, а может быть ничего» (у корня файловой системы нет родителя).
- `.expect("...")` разворачивает `Option`: если значение есть — возвращает его, если нет — **аварийно завершает программу с этим сообщением**. Для инструмента разработчика это нормально: лучше упасть с понятным текстом, чем продолжить с неверным путём.
- `Path` против `PathBuf` — та же пара, что `&str` против `String`: ссылка на путь против владеющей копии. Функция возвращает копию (`to_path_buf`), чтобы вызывающий код мог ею свободно распоряжаться.

### 3.3. Тип «никогда»: `fn fail(msg: &str) -> !`

```rust
fn fail(msg: &str) -> ! {
    eprintln!("error: {msg}");
    exit(1);
}
```

Возвращаемый тип `!` читается как «**никогда не возвращается**»: функция гарантированно завершает программу. Компилятор это знает, поэтому `fail(...)` можно использовать там, где ожидается любое значение:

```rust
let version = toml_value(&manifest, "version")
    .unwrap_or_else(|| fail("no version in Cargo.toml"));
```

- `eprintln!` печатает в **stderr** (поток ошибок), а не в stdout. Это дисциплина CLI: полезный вывод — в stdout, ошибки — в stderr, тогда вывод можно безопасно передавать по конвейеру (`cargo xtask status | grep version`).
- `exit(1)` — ненулевой код выхода, по нему скрипты и CI понимают, что команда провалилась.

### 3.4. Чтение значения из TOML: `toml_value`

```rust
fn toml_value(manifest: &str, key: &str) -> Option<String> {
    manifest.lines().find_map(|line| {
        let line = line.trim();
        line.strip_prefix(key)?
            .trim_start()
            .strip_prefix('=')?
            .trim()
            .strip_prefix('"')?
            .strip_suffix('"')
            .map(str::to_string)
    })
}
```

Это самая «плотная» функция, разберём по слоям.

**Задача:** найти в тексте `Cargo.toml` строку вида `version = "1.1.1"` и вытащить `1.1.1`.

- `.lines()` — итератор по строкам текста.
- `.find_map(|line| ...)` — пройти строки и вернуть **первый** не-`None` результат замыкания. Замыкание (`|line| { ... }`) — безымянная функция, создаваемая на месте.
- Цепочка `strip_prefix(...)?` — пошаговое «откусывание» от строки:
  - `"version = \"1.1.1\""` → `strip_prefix("version")` → `" = \"1.1.1\""`;
  - → `strip_prefix('=')` → `" \"1.1.1\""`;
  - → `strip_prefix('"')` → `"1.1.1\""`;
  - → `strip_suffix('"')` → `"1.1.1"`. Готово.
- Каждый `strip_prefix` возвращает `Option`: `Some(остаток)` если префикс был, `None` если нет. Знак `?` после него означает: «если `None` — сразу выйти из замыкания с `None`» (то есть «эта строка не подходит, идём к следующей»). Без `?` пришлось бы писать четыре вложенных `if let` — `?` схлопывает их в читаемую цепочку.

> **Почему не библиотека для TOML?** Полноценный парсер (`toml` crate) умеет всё, но нам нужно прочитать и переписать два конкретных значения, **не переформатировав весь файл** (парсер-сериализатор может переставить ключи и убрать комментарии). Построчная обработка проще, бережнее к файлу и не тянет зависимостей. Это инженерный компромисс, и он здесь осознанный: формат `Cargo.toml` нам полностью подконтролен.

### 3.5. Структура с методами: `struct Project`

```rust
struct Project {
    root: PathBuf,
    manifest: String,
    version: String,
    date: String,
}

impl Project {
    fn load() -> Self { ... }
    fn readme_line(&self) -> String { ... }
    fn readme_in_sync(&self) -> bool { ... }
}
```

- `struct` — просто группа полей (как dataclass в Python или объект в JS, но со строгими типами).
- `impl Project { ... }` — блок методов. В Rust данные и поведение объявляются раздельно.
- `fn load() -> Self` — **ассоциированная функция** без `self` (аналог статического метода/конструктора): вызывается как `Project::load()`.
- `fn readme_line(&self)` — метод, который **читает** данные структуры: `&self` означает «заимствую структуру на время вызова, не изменяя её». Если бы метод менял поля, было бы `&mut self`. Это центральная идея Rust — владение и заимствование: компилятор всегда знает, кто и как использует данные, и не даёт изменить их из двух мест одновременно.

Зачем структура вообще? Команды `status`, `sync`, `release` работают с одним и тем же набором данных (корень, манифест, версия, дата). Без структуры каждая функция принимала бы четыре параметра. Со структурой — один связный «снимок состояния проекта».

### 3.6. Запуск внешних программ: `Command`

```rust
fn git(root: &Path, args: &[&str]) -> Option<String> {
    let out = Command::new("git").args(args).current_dir(root).output().ok()?;
    out.status
        .success()
        .then(|| String::from_utf8_lossy(&out.stdout).trim().to_string())
}
```

- `Command::new("git").args(...).output()` — собрать и запустить процесс, дождаться завершения, забрать stdout/stderr. Аналог `subprocess.run` в Python.
- `.current_dir(root)` — выполнить в корне проекта, а не там, откуда запущен xtask.
- `.ok()?` — превратить `Result` (успех/ошибка запуска) в `Option` и выйти с `None`, если git вообще не запустился. То есть **отсутствие git — не катастрофа**: вызывающий код просто не покажет git-информацию.
- `.then(|| ...)` на `bool`: если `true` — вычислить значение и вернуть `Some(...)`, если `false` — `None`. Однострочный способ сказать «значение есть только при успехе».
- `String::from_utf8_lossy` — байты вывода → строка; невалидный UTF-8 заменяется символом `�` вместо паники («lossy» — «с потерями»).

> **Почему так правильно:** инструмент различает ошибки, которые **должны** останавливать работу (нет версии в Cargo.toml → `fail`), и ситуации, которые допустимы (нет git → `Option::None`). Это и есть культура обработки ошибок в Rust: `Result`/`Option` заставляют в каждой точке решить, что делать с неудачей, — забыть проверить ошибку, как в bash, невозможно.

### 3.7. Команда `release`: цикл по командам git

```rust
for args in [
    vec!["add", "Cargo.toml", "Cargo.lock", "README.md"],
    vec!["commit", "-m", &format!("Release {tag}")],
    vec!["tag", &tag],
    vec!["push"],
    vec!["push", "origin", &tag],
] {
    let status = Command::new("git")
        .args(&args)
        .current_dir(root)
        .status()
        .expect("failed to run git");
    if !status.success() {
        fail(&format!("git {} failed", args.join(" ")));
    }
}
```

(`.status()` в отличие от `.output()` не перехватывает вывод — git печатает прямо в терминал, и видно, что происходит.)

Последовательность git-команд оформлена **данными** (массив списков аргументов), а исполнение — одним циклом с единообразной проверкой ошибок. Если бы каждая команда вызывалась отдельным блоком кода, проверку `success()` пришлось бы копировать пять раз — а копипаста рано или поздно расходится.

Обратите внимание: `release` добавляет в коммит **только три файла**, а не `git add -A`. Это защита: случайные незакоммиченные правки не «приклеятся» к релизному коммиту.

## 4. Почему вся схема устроена именно так

**Единый источник правды.** Версия живёт в одном месте — `Cargo.toml`. Все остальные упоминания (README, футер UI, лог) — производные: либо генерируются при сборке (`env!("CARGO_PKG_VERSION")` — версия зашивается в бинарник компилятором), либо синхронизируются инструментом. Когда данные дублируются вручную, они **обязательно** разъедутся; здесь рассинхрон либо невозможен, либо обнаруживается командой `status`.

**Изменения только через инструмент.** Человек, правящий версию руками, забудет про дату, README или Cargo.lock. Инструмент не забывает. Та же причина, по которой существуют миграции БД, а не «поправь схему руками».

**Fail fast.** Любая проблема (битая версия, упавший git) останавливает работу с внятным сообщением — вместо того чтобы молча сделать половину работы. Половина релиза хуже, чем отсутствие релиза.

**Дешёвая проверяемость.** `cargo xtask status` отвечает на вопрос «всё ли синхронно?» за секунду. Проверка, которую дёшево запустить, — это проверка, которую реально запускают.

## 5. Мини-упражнение: добавьте свою команду

Попробуйте добавить команду `cargo xtask tag`, которая печатает имя тега для текущей версии (`v1.1.1`). Нужно две правки в `xtask/src/main.rs`:

1. Функция:

```rust
fn show_tag() {
    let project = Project::load();
    println!("v{}", project.version);
}
```

2. Ветка в `match` внутри `main` (и строчка в `print_help`):

```rust
["tag"] => show_tag(),
```

Запуск: `cargo xtask tag`. Если компилятор ругается — читайте сообщение целиком: ошибки rustc подробные и почти всегда содержат подсказку «попробуйте написать так». Это лучший учитель языка из доступных.
